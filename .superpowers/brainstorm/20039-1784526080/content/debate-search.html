🔥 附加议题：是否需要全局搜索？

创始人提出一个问题：除了场景化筛选，是否需要全局模糊搜索？

📊 产品经理 权衡利弊

搜索和筛选解决的是不同问题，两者都需要，但粒度不同。

筛选：用户模糊记得类型/感觉 → 「我收藏过的咖啡店有哪些」
搜索：用户精确记得名字/关键词 → 「那家叫什么的串串店」

但我们要问：足迹手帐的用户体量需要全局搜索吗？
如果只有 30-50 个地点，翻翻卡片就能找到，搜索反而是过度设计。
如果用户有 200+ 个地点，筛选也不够了，必须有搜索。

我建议现在就加，但要轻量。不是独立搜索页，而是首页顶部一个搜索图标，点击展开搜索框。

🧑‍💻 用户代表 强烈需要

​绝对需要。

我的地点不多，大概四十几个。但我经常遇到这个场景：
「上次和朋友去的那家店，名字里有个什么茶...什么山...记不清了」

我可以搜「茶」或者「山」，模糊匹配。
而不是把四十多张卡片一张一张翻。

搜索解决的是「我知道一点、但不全知道」的问题。
这是筛选做不到的。

🎨 设计师 交互方案

不建议独立搜索页。那太重了，而且和「手帐」的感觉不符。

建议：轻量内嵌搜索。
· 首页顶部筛选栏右侧，一个搜索图标
· 点击后筛选栏收起，搜索框展开
· 输入时实时过滤下方卡片
· 搜索范围：地点名称 + 标签名 → 模糊匹配
· 取消搜索 / 清空 → 恢复原始列表

交互轻，不跳页，不破坏浏览体验。

⚙️ 工程师 技术评估

很简单。

后端接口按名称模糊搜索 → 已有或新增，一行 SQL LIKE 的事。
前端实时过滤 → 如果数据量小，可以直接前端过滤，都不用请求后端。
难度：极低。半小时能做好。

唯一需要决策的：搜索范围要不要扩展到「打卡内容」？
比如用户在打卡里写了「这家店的拿铁很好喝」，搜「拿铁」能不能搜到？
现在只搜地点名和标签就够了，打卡内容搜索可以后续迭代。

🚀 运营增长 不反对

搜索对增长不是核心杠杆，但不反对。

唯一提醒：别让搜索入口太抢视觉。首页的主路径是浏览和筛选，搜索是辅助。一个搜索图标就够了，不要大搜索框占位置。

🔑 结论

结论：需要，但做轻量内嵌搜索，不做独立搜索页。

· 位置：首页筛选栏右侧，搜索图标
· 交互：点击展开搜索框，实时过滤卡片，不跳页
· 范围：地点名称 + 标签 → 模糊匹配
· 技术：极低难度
· 后续扩展：可扩展到打卡内容搜索

轻量内嵌搜索 — 需要，但别太重。你认可吗？
