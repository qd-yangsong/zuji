🔥 第六轮：首页该怎么筛选和查找？

问题 ⑦ — 按年/日期筛选价值有限，用户更习惯按喜好、用途、场景查找

💥 创始人对现状的判断

首页按年、按日期的标签，当用户收藏地点越来越多后，这种筛选价值有限。
用户更习惯按喜好、用途、场景去查找。

🧑‍💻 用户代表 真实使用场景

来说说我真实需要查的场景：

场景一：朋友问我「有什么好吃的推荐？」→ 我想快速找到我有标记「好吃」的店
场景二：这周末想带女朋友去个有氛围的地方 → 我想找「约会」、「氛围感」的地点
场景三：想找一个安静的地方看书 → 我想找「安静」、「适合独处」的地儿
场景四：上次去成都的那家串串店叫什么来着？→ 我想找「成都」相关的

你看，我什么时候会说「我想找2024年3月收藏的地点」？几乎不会。

用户找东西是按场景、按用途、按感觉，很少按时间。时间排序是数据库的逻辑，不是人的逻辑。

📊 产品经理 拆解问题

用户代表说得对。但我需要拆一下：「筛选」和「搜索」是两回事。

筛选 = 选一个维度，看到符合条件的卡片
搜索 = 输入关键词，精确匹配

用户代表的四个场景里，场景一/二/三是筛选，场景四（成都的串串店）更接近搜索。

现在的标签系统本身就是按「喜好」来筛选的 — 咖啡店、餐厅、书店...
但问题出在标签不够「表达力」。

现在的标签是：咖啡店、餐厅、公园...（客观分类）
用户需要的是：「氛围超好」、「适合约会」、「我反复去的」...（主观感受）

我建议两层筛选维度：
顶层 — 快捷筛选：「全部」「我爱去的」「想再去」「有推荐的」— 情感化分类
展开层 — 标签筛选：现在的标签系统，但增加「情感标签」（氛围好 / 适合约会 / 性价比高...）

🎨 设计师 交互方案

我来画一下首页的筛选栏设计：

顶部一排横向滚动的筛选词：
「全部」「常去的」「想再去」「餐厅」「咖啡店」「出去玩」「探店」

这不是死板的分类，而是用户会说的话。每个词像一个「小标签」，点击就过滤。

关键交互：
· 默认选中「全部」，展示所有卡片
· 卡片按最近打卡时间倒序展示（用户最常用这种行为）
· 用户可以通过筛选词快速缩小范围
· 不展开时只有一行，不占屏幕空间

另外，如果用户数量多，可以考虑加一个轻微的搜索图标，点击可以输入地点名精确搜索。

时间相关的排序怎么办？保留，但不作为主要入口。可以放在一个「排序」按钮里 — 默认最近使用，可以切换到「按收藏时间」「按区域」等。

🚀 运营增长 从听到的思考

我有个更大胆的想法：标签筛选词应该是智能生成的。

不是我们预设的固定分类，而是根据用户实际收藏的地点类型，动态生成。

如果用户收藏了很多咖啡店 → 「咖啡店」标签排在前面
如果用户最近常去餐厅 → 「餐厅」标签自动升到前面
如果用户去了一个城市旅游 → 自动出现城市标签

这样筛选栏是为用户定制的、会变化的、有生命力的。

但我知道工程师会告诉我这个实现成本高。那就分两步走：
第一步：固定一组筛选词，覆盖主流场景
第二步：用户收藏多了以后，智能调整排序和出现哪些词

⚙️ 工程师 评估

运营你想得太远了，但我不反对这个方向。

标签筛选（固定词）：
· 后端按标签字段过滤 → 已有接口，加 query 参数即可
· 前端横向滚动标签栏 → 小程序标准组件，无技术难度
难度：极低。

搜索：
· 后端按名称模糊搜索 → 已有或容易加
· 前端搜索框 → 标准交互
难度：低。

智能排序/动态标签：
· 需要统计用户的标签使用频率 → 新接口
· 排序逻辑在前端即可完成
难度：中。可以做，但不建议第一版就上。

另外，我注意到现在首页还有「按年筛选」。我的建议是：别删。把它从顶部的横向标签移到排序选项里，用户偶尔也会用到的，只是不作为主入口。

🔑 本轮共识

筛选栏改版：从「按年份横向标签」→「按场景/类型的横向筛选词」
筛选词建议：「全部」「常去的」「想再去」「餐厅」「咖啡店」「出去玩」「探店」等用户语言的词
默认排序：最近打卡/收藏时间倒序（用户最常用）
保留时间筛选：移到排序选项里，作为可用但非默认的维度
搜索：增加搜索入口，支持模糊搜索地点名
远期方向：标签词根据用户数据智能排序和生成

「筛选词用场景化语言，时间筛选退到排序里」这个方向，你认可吗？
