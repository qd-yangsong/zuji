# 足迹手帐 · 产品设计文档（Spec）

> 创建日期：2026-06-17  
> 版本：v0.1（头脑风暴结论稿）  
> 形态：微信小程序

---

## 一、产品定位

**一句话定义**：
一款帮助用户**收藏对自己重要的地点 & 与之关联的记忆**的微信小程序。地图是底层，卡片是表达，打卡是再次记录的仪式。

**核心理念**：
> 地点 = 记忆的容器。每一次打卡都是给容器里多放一颗珍珠。

**与现有产品的差异化**：
- 不是地图工具（不和高德/百度/大众点评在导航、推荐上正面竞争）
- 不是社交工具（不主打陌生人 / 好友足迹流）
- 是「**情感型记忆收藏品**」，气质趣味年轻、有仪式感

---

## 二、目标用户

通用型工具，覆盖：
- 热爱旅行的年轻人（18-35 岁），记录旅行足迹
- 注重生活仪式感的白领，记录日常发现（美食、咖啡馆等）
- 家庭用户，记录家庭出行和亲子活动地点

---

## 三、核心痛点

1. **散落问题**：日常地点收藏散落在大众点评、高德、百度地图等不同应用，无法聚合
2. **难找回**：原生地图应用的「收藏」功能弱，UI 简陋，无法记录细节
3. **无情感**：没有产品把「重复到访」「与地点的记忆」做成可沉淀的内容
4. **分享麻烦**：朋友问推荐时，从多平台翻找极不便利

---

## 四、核心价值主张

```
聚合散落的地点收藏 → 用自己的方式记录 → 通过打卡持续沉淀 → 在需要时找回 / 分享
```

## 五、产品架构

### 5.1 主入口（关键差异点：不是地图，是卡片墙）

```
[首页 = 收藏卡片墙]
├── 顶部：搜索框 + 视图切换（卡片 / 地图 / 时间轴）
├── 分类标签：全部 | 美食 | 旅行 | 咖啡 | 自定义...
├── 排序：最近添加 | 按年 | 按日期 | 打卡次数
└── 卡片流（瀑布流 / 双列）：
     每张卡片 = 封面图 + 自定义昵称 + 地点真实名 + 打卡次数 + 最近一次记录摘要
```

### 5.2 五大核心模块

| 模块 | 功能描述 |
|---|---|
| 🗂️ 卡片墙（首页） | 所有收藏地点的「记忆相册」，多维度排序与筛选 |
| 📍 地点详情 | 一个地点的完整档案：基本信息 + 多次打卡记录的时间轴 |
| ✍️ 打卡动作 | 自动定位匹配 → 录入图文 + 事件标签 → 沉淀到该地点的时间轴 |
| 🗺️ 旅程地图 | 由打卡行为自动生成的可视化地图（按时间 / 地点聚合） |
| 🏷️ 标签体系 | 三层标签（属性 / 场景 / 事件），支持自定义和编组，是「快速找地」「场景化推荐」的核心 |

### 5.3 「快速找地」入口（首页顶部模块）

> 解决最高频复看 / 分享场景：「带孩子去哪好？」「招待外地朋友去哪好？」

```
🎯 快速找地（基于场景标签的快捷入口）
┌──────────────────────────────────────────┐
│ 🧒 带孩子    🤝 招待客人    📅 周末      │
│ 👨‍👩‍👧 家庭聚餐  🍻 朋友小聚   ➕ 自定义   │
└──────────────────────────────────────────┘
点击 → 直接进入该标签下所有地点的卡片视图
```

### 5.4 地点详情页结构（核心页面）

```
[封面图]
[自定义昵称（大字）]      ← 用户取的名字，如"我家楼下深夜小馆"
[地点真实名 + 地址]       ← 灰色小字
[属性标签：美食 / 深夜 / 安静]              ← Place.attribute_tags
[场景标签：适合朋友小聚 / 适合深夜聚会]     ← Place.scene_tags
[打卡 N 次 · 收藏于 2024.03.15]
─────────────────
[一键打卡按钮]            ← 显眼 CTA
─────────────────
[记忆时间轴]
  ▸ 2024.05.20 第3次打卡 · #同学聚会
       "和老张又来了，加了个新菜..." 📷📷📷
  ▸ 2024.04.10 第2次打卡 · #剧本杀
       文字日记
  ▸ 2024.03.15 首次收藏 · #朋友推荐
       文字日记
─────────────────
[底部小地图缩略]（点击展开全屏地图）
[分享按钮]
```

## 六、数据模型

### 6.1 三层标签体系（关键架构）

```
┌─────────────────────────────────────────────────┐
│                  三层标签体系                    │
├─────────────────────────────────────────────────┤
│  ① 地点属性标签（贴在 Place 上）                │
│     例：美食 / 咖啡馆 / 网红 / 安静 / 价位贵     │
│                                                 │
│  ② 地点场景标签（贴在 Place 上，可编组）        │
│     例：适合带孩子 / 适合招待客人 / 适合周末     │
│     ★ 支持用户自定义标签 + 自定义分组            │
│                                                 │
│  ③ 打卡事件标签（贴在 CheckIn 上）              │
│     例：同学聚会 / 单位年会 / 剧本杀 / 家长会    │
│     ★ 是远期「记忆场景 / 足迹小说」的核心数据    │
└─────────────────────────────────────────────────┘
```

**为什么必须分三层？**
- 属性 = 地点是什么样的（恒定）
- 场景 = 我在什么情况下会想到这里（用于「快速找地」）
- 事件 = 这次到访发生了什么（用于回忆 / 关系 / 故事）
- 三者混用会导致数据语义混乱、检索失效

### 6.2 系统预设标签（开箱即用，用户可删改）

| 类型 | 默认值示例 |
|---|---|
| 属性标签 | 美食、咖啡、酒吧、书店、景点、酒店、博物馆、公园、商场、办公地、学校 |
| 场景标签「陪伴场景」组 | 适合带孩子、适合带父母、适合带宠物 |
| 场景标签「社交场景」组 | 适合朋友小聚、适合招待重要客人、适合接待外地朋友、适合相亲约会、适合团建 |
| 场景标签「时间场景」组 | 适合日常去、适合周末、适合长假、适合工作日中午、适合深夜 |
| 场景标签「心情场景」组 | 适合一个人静一静、适合发呆、适合赶 deadline |
| 事件标签 | 同学聚会、单位年会、家庭聚餐、家长会、剧本杀、密室、看演出、看展、度假、出差、约会、生日庆祝、纪念日、闺蜜局、兄弟局 |

### 6.3 实体设计

```
Place（地点）
  - id
  - 真实名称 / 自定义昵称
  - 经纬度 / 地址
  - 封面图
  - 收藏时间
  - 打卡次数（计算字段）
  - attribute_tag_ids[]  ← 属性标签（多选）
  - scene_tag_ids[]      ← 场景标签（多选）

CheckIn（打卡记录）← 一个 Place 对应 N 条 CheckIn
  - id
  - place_id
  - 时间
  - 文字日记
  - 图片列表
  - 是否首次收藏（true / false）
  - event_tag_ids[]      ← 事件标签（多选）★ 远期内容生成的关键数据

Tag（标签）
  - id
  - 名称
  - 类型（attribute / scene / event）
  - group_id（可空，仅 scene 标签可归组）
  - is_system（是否系统预设）
  - usage_count（使用次数，用于热门排序）

TagGroup（标签组）
  - id
  - 名称（如"陪伴场景"）
  - 颜色 / 图标
  - tag_type（限定该组只能容纳的标签类型，目前仅 scene）
  - is_system（是否系统预设）

Collection（地点合集）← 用户自定义打包多个 Place
  - id
  - 合集名称
  - 封面
  - place_ids[]
  - 描述

JourneyMap（旅程地图）← 由 CheckIn 自动聚合生成（视图层概念）
```

### 6.4 地图渲染数据源拆分（架构师重要建议）

> 历史教训：marker 列表会被业务渲染覆盖，搜索仅遍历当前 markers 会失败。

- `baseMarkers`：用户所有收藏地点 marker
- `journeyMarkers`：旅程地图模式下的打卡轨迹 marker
- 最终 `map.markers = merge(baseMarkers, journeyMarkers)`，按当前视图模式取并集
- 搜索维度必须覆盖：名称 + 自定义昵称 + 地址 + 属性标签 + 场景标签 + 事件标签 + 日记内容

## 七、MVP 范围

### P0（第一版必做）
- ✅ 地图选点 / 当前定位收藏地点
- ✅ 自定义昵称 + 封面图
- ✅ 三层标签体系（属性 / 场景 / 事件），系统预设 + 用户自定义 + 场景标签可编组
- ✅ 卡片墙首页（排序：最近添加 / 按年 / 按日期 / 打卡次数）
- ✅ 「快速找地」入口（场景标签快捷筛选）
- ✅ 卡片 / 地图 / 时间轴 三视图切换
- ✅ 地点详情 + 一键打卡（图文 + 事件标签）
- ✅ 记忆时间轴（一个地点的多次打卡按时间排列，显示事件标签）
- ✅ 旅程地图（打卡轨迹可视化）
- ✅ 地点合集（创建 / 编辑）
- ✅ 单点分享 + 合集分享（生成卡片，对方无需安装可跳转地图导航）

### P1（第二版）
- 事件回顾视图（按事件标签聚合，如"我所有的同学聚会发生在哪")
- 关系记忆（@朋友 标签，配合事件标签生成关系画像）
- 标签使用情况统计 / 热门标签
- 多视图切换交互优化
- 数据备份与导出

### P2（远期 · 内容生成与情感唤起）
- 语音 / 视频日记
- 附近提醒（到达曾打卡地点附近）
- 年度回顾 / 纪念日提醒
- 第三方平台数据导入（大众点评、高德、百度）
- ★ 「记忆场景」专题页（基于事件标签自动生成，如"2025 春节假期"）
- ★ 「足迹小说」AI 串联打卡记录生成图文故事
- ★ 场景化导出（如"2024 年家庭聚餐合集"打包成手帐 PDF）

## 八、分享方案（关键技术决策）

> 历史教训：wx.openLocation 仅能打开地图到某点，无法多路线 / 路况，不能作为主导航方案。

- 主方案：**生成微信小程序卡片 + H5 链接**
  - 朋友点击卡片 / 链接，打开预览页面（无需注册或安装）
  - 预览页面显示：地点信息 + 自定义昵称 + 用户记忆摘要
  - 提供「跳转高德 / 百度 / 苹果地图导航」按钮
- 兜底方案：`wx.openLocation` 跳第三方地图

## 九、产品气质（视觉与交互）

**风格定位**：趣味年轻 + 仪式感

- **视觉**：插画驱动、动效丰富、年轻化配色
- **文案温度**：避免"我的地点""收藏列表"，使用"我的记忆地图""又一次来到这里"
- **空状态插画**：引导式（如"还没有任何回忆，去标记第一个对你重要的地方吧"）
- **打卡仪式感**：打卡时有动画 / 微交互（如撕贴纸、盖章音效），让"打卡"变成有情绪的动作
- **首次收藏 vs 重复打卡**：UI 区分，重复打卡有"故地重游"的提示

## 十一、风险与挑战

### 11.1 留存风险（运营提示）
- 仪式感产品长期留存难，需设计被动唤起机制（P2 阶段实现：附近提醒、年度回顾、纪念日提醒）

### 11.2 冷启动风险
- 用户从零录入，前期空白期体验差
- 缓解：MVP 提供精美空状态插画 + 引导教程，鼓励首次收藏

### 11.3 技术风险（架构师提示）
- 微信小程序地图组件 API 受限，复杂交互（多 marker 聚合、自定义图层）有性能瓶颈
- 图片存储成本：需规划云存储（建议初期用微信云开发，控制单用户配额）
- 数据源拆分必须严格执行（baseMarkers / journeyMarkers 不可混用）

### 11.4 商业化（暂不考虑）
- 早期不考虑商业化，先验证 PMF
- 远期可考虑：高级会员（云存储扩容、年度回顾电子书等增值服务）
